# NATS TLS Certificates

TLS certificates for NATS mTLS authentication (ADR-0018, ADR-0030) are issued
directly from Vault PKI (`pki_int`) at runtime. No certificate files are kept
in the repository tree. No mkcert anywhere.

## How It Works

1. **At container startup**, the `nats-init` one-shot container:
   - AppRole-logs in as `foundation-agent-ruby-core-nats-server`
   - Issues a fresh server cert from `pki_int/issue/ruby-core-nats-server`
   - Writes `server-cert.pem`, `server-key.pem`, `ca.pem` to a Docker-managed
     named volume that NATS mounts read-only
   - Generates `auth.conf` from NKEY public keys read from Vault KV
2. **NATS** reads the cert + auth.conf from the shared volume and starts.
3. **The `nats-cert-renewer` sidecar** (long-running Vault Agent) re-issues
   the server cert at TTL/2 and SIGHUPs NATS to reload TLS in-place. Existing
   client connections survive — only new TLS handshakes use the rotated cert.
4. **The 5 service containers** (gateway, engine, notifier, presence,
   audit-sink) each AppRole-log in via `pkg/boot/pki.go` and issue their own
   client cert from `pki_int/issue/ruby-core-<svc>`. An in-process goroutine
   renews at TTL/2.
5. **The smoke test** (run by `deploy-staging.sh` / `deploy-prod.sh`)
   AppRole-logs in as admin and issues a short-lived (1h) cert from
   `pki_int/issue/ruby-core-admin` for each run.

## Vault paths

**PKI roles** (issuance source, defined in foundation):

| Role | Used by |
|------|---------|
| `pki_int/roles/ruby-core-nats-server` | nats-init + nats-cert-renewer |
| `pki_int/roles/ruby-core-{gateway,engine,notifier,presence,audit-sink}` | the 5 services via `boot.go` |
| `pki_int/roles/ruby-core-admin` | smoke-test.sh |

**NKEY seeds** (still in KV; NKEY auth is orthogonal to TLS):

| Path | Contents |
|------|----------|
| `secret/ruby-core/nats/<service>` | NKEY seed + public key (for auth.conf generation) |

## Commands

```bash
# Generate NKEY seeds (one-time per fresh host)
make setup-creds

# Force regeneration of NKEY seeds
make setup-creds-force
```

TLS material is never generated by Make targets — it's all issued at runtime
from PKI. See `scripts/fetch-nats-certs.sh`, `pkg/boot/pki.go`, and
`scripts/smoke-test.sh` for the issuance code paths.

## Security notes

- No certificate private keys are stored on disk in the repo, in Vault KV,
  or in any persisted Docker volume that survives a container restart cycle.
  Each cert lives only in the issuing service's memory (Go services) or in
  an ephemeral Docker named volume that's repopulated on every nats-init run
  (NATS server cert).
- Certificate rotation is automatic at TTL/2 — no operator action required.
- Revocation: `make revoke-client-cert SERIAL=<colon-hex>` on the foundation
  side, propagated to NATS via the CRL refresh path.
- See ADR-0030 for the direct-PKI architecture decision; ADR-0008 (foundation)
  for the broader Vault Agent secret-distribution pattern.
